# データベース定義書

## アプリ名

AI海亀のスープ

## データベース名

umigame

## 概要

本アプリは「ウミガメのスープ」をAIがゲームマスターとして1人プレイできるWebアプリである。データベースはユーザー情報、ユーザーが投稿した問題（真相とAI生成の問題文）、問題のタグ、プレイ履歴の4テーブルで構成される。

---

## テーブル一覧

| テーブル名 | 役割 |
| --- | --- |
| users | アプリを利用するユーザーの情報を管理する |
| problems | ユーザーが投稿した真相と、AIが生成した問題文を管理する |
| problem_tags | 問題に紐づくジャンルタグ（1問題につき最大3つ）を管理する |
| plays | ユーザーが問題をプレイした履歴を管理する |

---

## 1. users（ユーザーテーブル）

アプリにログインするユーザーの情報を保持する。

| カラム名 | 型 | NULL | キー | 説明 |
| --- | --- | --- | --- | --- |
| id | INT UNSIGNED | NOT NULL | PK（自動採番） | ユーザーID |
| name | VARCHAR(100) | NOT NULL | - | ユーザー名 |
| email | VARCHAR(255) | NOT NULL | UNIQUE | メールアドレス（ログインに使用） |
| password | VARCHAR(255) | NOT NULL | - | パスワード（暗号化して保存） |
| created_at | DATETIME | NOT NULL | - | 登録日時 |

---

## 2. problems（問題テーブル）

ユーザーが投稿した真相と、それをもとにAIが生成した問題文を保持する。

| カラム名 | 型 | NULL | キー | 説明 |
| --- | --- | --- | --- | --- |
| id | INT UNSIGNED | NOT NULL | PK（自動採番） | 問題ID |
| user_id | INT UNSIGNED | NULL許容 | FK → users.id | 投稿したユーザーのID |
| story | TEXT | NOT NULL | - | ユーザーが入力した真相 |
| question_text | TEXT | NOT NULL | - | AIが自動生成した問題文 |
| difficulty | ENUM('easy','normal','hard') | NOT NULL | - | 難易度（デフォルト: normal） |
| created_at | DATETIME | NOT NULL | - | 投稿日時 |

**外部キー制約**：`user_id` は `users.id` を参照し、ユーザー削除時、投稿者情報がNULLになる（問題自体は残る）

---

## 3. problem_tags（問題タグテーブル）

1つの問題に対して複数のジャンルタグ（最大3つ）を持たせるための中間テーブル。

| カラム名 | 型 | NULL | キー | 説明 |
| --- | --- | --- | --- | --- |
| problem_id | INT UNSIGNED | NOT NULL | PK（複合）／FK → problems.id | 問題ID |
| tag | 15種類のENUM | NOT NULL | PK（複合） | ジャンルタグ（15種類から選択、1問題につき最大3行） |

**外部キー制約**：`problem_id` は `problems.id` を参照し、問題削除時、紐づくタグも削除される（ON DELETE CASCADE）

**補足**：1問題あたり最大3タグという制約はアプリケーション側（投稿画面）で担保し、DB側では複合主キーにより同一タグの重複登録のみを防ぐ

---

## 4. plays（プレイ履歴テーブル）

ユーザーが問題をクリアした履歴を保持する。クリア・ギブアップ時にレコードを作成する方針とする（途中再開機能は本バージョンでは対象外）。

| カラム名 | 型 | NULL | キー | 説明 |
| --- | --- | --- | --- | --- |
| id | INT UNSIGNED | NOT NULL | PK（自動採番） | プレイID |
| user_id | INT UNSIGNED | NULL許容 | FK → users.id | プレイしたユーザーのID（未ログインの場合はNULL） |
| problem_id | INT UNSIGNED | NULL許容 | FK → problems.id | プレイした問題のID |
| question_count | INT UNSIGNED | NOT NULL | - | 質問した回数（デフォルト: 0） |
| clear_time_sec | INT UNSIGNED | NULL許容 | - | クリアにかかった時間（秒） |
| is_cleared | TINYINT(1) | NOT NULL | - | クリアしたか（0: 未クリア, 1: クリア） |
| created_at | DATETIME | NOT NULL | - | プレイ日時 |

**外部キー制約**：
・`user_id` → `users.id` を参照（未ログインプレイの場合はNULL）。ユーザー削除時、そのユーザーのプレイ履歴も削除される（ON DELETE CASCADE）
・`problem_id` → `problems.id` を参照。問題削除時、プレイ履歴は残り、problem_idがNULLになる（ON DELETE SET NULL）

---

## テーブル間の関係

```
users (1) ──── (多) problems
  └─ 1人のユーザーは複数の問題を投稿できる

users (1) ──── (多) plays
  └─ 1人のユーザーは複数のプレイ履歴を持つ

problems (1) ──── (多) plays
  └─ 1つの問題は複数回プレイされる

problems (1) ──── (多) problem_tags
  └─ 1つの問題は最大3つのタグを持つ
```

---

## 設計方針・補足

- 質問ログ（チャット履歴）はDBに保存せず、フロントエンド側のセッション内で保持し、プレイ終了時に破棄する方針とした。これはDB肥大化を避けるためであり、ヒント生成にはセッション内の履歴で十分対応できる。
- 開発環境ではローカルのMySQLを使用する。将来的にクラウドDB（PlanetScale / Supabase等）への移行を予定しているが、SQL系DBを使い続けることで移行コストを抑える設計とした。
- パスワードはbcryptjsで暗号化して保存する
- タグは15種類のENUMで管理し、アプリ側で日本語表示に変換する。1問題につき最大3つまで選択でき、`problem_tags`テーブルで多対多を表現する
- ギブアップ時もplaysにレコードを作成し、is_clearedで0（ギブアップ）/1（クリア）を管理する
- 未ログインでもプレイ可能とし、その場合`plays.user_id`はNULLとする
