# 基本設計書

## アプリ名

AI海亀のスープ

## 1. システム概要

「ウミガメのスープ」（水平思考パズル）をAIがゲームマスターとして進行する、1人用のWebアプリ。プレイヤーは提示された問題文に対してyes/noで答えられる質問を重ね、真相（トリック）を推理する。真相の投稿・問題文の自動生成・質問への回答・正解判定を、いずれもAIが担当する。

ログインしなくても全機能を利用できる（ゲストプレイ）。ログインすると、自分がクリア済みの問題が一覧画面でわかるようになる。

---

## 2. 機能一覧

| 機能 | 概要 |
| --- | --- |
| 問題の投稿 | ユーザーが真相を入力すると、AIが問題文と正解判定用の要点を自動生成する |
| プレイ（質問） | チャット形式で質問すると、AIが真相に照らして「はい/いいえ/関係ない」を判定する |
| ヒント | AIが真相を明かさない範囲で、次に何を質問すべきかのヒントを出す |
| 真相を答える | 専用のポップアップから最終回答を送信し、AIが正解かどうかを判定する |
| ギブアップ | プレイヤーが自らギブアップするか、質問回数の上限に達すると自動でギブアップになる |
| 問題一覧 | 投稿された問題を一覧表示（プレイ数順）。ランダムに1問選ぶボタンもある |
| 結果表示 | 質問数・クリア時間・難易度・真相・正解の要点を表示する |
| 会員登録・ログイン・ログアウト | メールアドレスとパスワードでアカウントを作成・認証する |

---

## 3. 画面一覧・画面遷移

| 画面 | パス | 概要 |
| --- | --- | --- |
| タイトル画面 | `/` | ゲーム開始・投稿・一覧への入口。ログイン状態に応じて表示が変わる |
| ゲーム画面 | `/play/[id]` | 問題文表示＋チャットUI。質問・ヒント・真相を答える・ギブアップの起点 |
| 正解・解説画面 | `/result/[id]` | プレイ結果（正解／ギブアップ）と真相・要点を表示 |
| 問題投稿画面 | `/post` | 真相・難易度・タグを入力し、AIに問題を生成させる |
| 問題一覧画面 | `/problems` | 投稿済みの問題をカード形式で一覧表示 |
| ログイン画面 | `/login` | メールアドレス・パスワードでログイン |
| 新規登録画面 | `/signup` | 名前・メールアドレス・パスワードでアカウント作成 |

```
タイトル画面
 ├─ 今日の問題を解く → ゲーム画面
 ├─ 問題を投稿する   → 投稿画面 → (AI生成後) → ゲーム画面
 ├─ 問題一覧を見る   → 問題一覧画面 → ゲーム画面
 └─ ログイン / 新規登録 → ログイン画面 ⇄ 新規登録画面 → タイトル画面

ゲーム画面
 ├─ 正解 → 正解・解説画面
 └─ ギブアップ（自動 or 手動） → 正解・解説画面

正解・解説画面
 ├─ 次の問題へ → ゲーム画面
 └─ 問題一覧に戻る → 問題一覧画面
```

すべての画面から、タイトル画面へ戻るボタン（ホームアイコン）を設置している。

## 6. 主要な処理フロー

### 6.1 問題投稿フロー

1. ユーザーが真相・難易度・タグ（最大3つ）を入力して送信
2. サーバーアクションがGemini APIに真相を渡し、「問題文」と「正解判定用の要点（2〜4個）」をJSON形式で生成させる
3. `problems`テーブルに真相・問題文・要点・難易度を保存し、`problem_tags`テーブルにタグを保存
4. 生成された問題のゲーム画面へ遷移

### 6.2 プレイフロー

1. ゲーム画面で問題文を表示し、プレイヤーが質問を入力
2. サーバーアクションがGemini APIに「問題文・真相・これまでのやり取り・新しい質問」を渡し、「はい/いいえ/関係ない」を判定させる
3. プレイヤーが「真相を答える」ポップアップから最終回答を送信すると、Gemini APIが正解判定用の要点を参考に「真相の核心を捉えているか」を判定する（要点を一言一句すべて満たす必要はない）
4. 正解と判定されれば`plays`にクリア記録を保存し、正解・解説画面へ遷移
5. 質問回数が難易度ごとの上限に達するか、プレイヤーが自らギブアップを選ぶと、`plays`にギブアップ記録を保存し、正解・解説画面へ遷移

### 6.3 認証フロー

1. 新規登録・ログイン時、サーバーアクションでパスワードをbcryptで検証（登録時はハッシュ化して保存）
2. 認証成功時、ユーザーIDを含むJWTをjoseで署名し、httpOnly Cookieにセッションとして保存
3. 各ページ・サーバーアクションはCookieのセッションを検証してログイン中ユーザーを取得する
4. ログインしていない場合はゲストとして扱い、`problems.user_id`・`plays.user_id`はNULLになる

## 7. ゲームルール

- 質問回数の上限は難易度ごとに固定：やさしい15回・ふつう20回・むずかしい30回
- 「質問する」「真相を答える」のどちらの発言も同じ回数としてカウントする
- 上限に達する、またはプレイヤーが自らギブアップを選ぶと、自動的にギブアップ扱いとなり結果が記録される
- 正解判定は「正解の要点」を参考にしつつ、真相の核心的なトリック・理由を的確に捉えていれば正解とする（細部の言い回しの違いは許容する）

---

## 8. 開発時の工夫

- `.env.local`の`MOCK_AI`フラグにより、Gemini APIを呼ばずに仮の応答を返す「モックモード」を用意した。開発・動作確認のたびにAPIの無料枠を消費しないための対策。
